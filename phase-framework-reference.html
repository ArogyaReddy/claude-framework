The Five Phases

Every non-trivial feature follows this sequence. Every time.

The quality of what Claude Code produces is a direct function of the structure you impose on the session. Unstructured sessions drift. They overengineer. They build the wrong thing confidently. They produce code that works today and is incomprehensible in three months.

The five-phase workflow exists to prevent all of that — not by constraining Claude, but by channeling it. Each phase has a single job. Each phase has a clear exit condition. Claude knows which mode it is in and behaves accordingly.

The Non-Negotiable Never build during Phase 1. Never skip Phase 2 for multi-file features. Never batch multiple chunks without committing. Never close a session with meaningful work without running /update. These four rules prevent 80% of the problems that come from working with Claude Code at scale.
02

Tools — What,
When, Why

Commands, hooks, agents, and skills. Each has a specific job.

Four tool types are available in Claude Code. Most guides treat them as interchangeable. They are not. Each is designed for a different class of problem. Using the wrong tool is not just inefficient — it produces worse outcomes.

Tool Type What It Is Triggered Best For Not For
Command
/plan /spec /chunk /verify /update		 Prompt template stored as .md in .claude/commands/. Claude reads and follows it. Manual — you type /name Structured phase transitions. Workflows you consciously initiate. Multi-step processes. Things that should always happen automatically. Independent review.
Hook
post-edit.sh pre-commit.sh		 Shell script that runs automatically when a trigger condition fires. Automatic — fires on event Quality gates that must ALWAYS run. Zero-discipline automation. Catching errors before they accumulate. Reasoning tasks. Anything needing Claude's judgment. Tasks with variable relevance.
Agent
/agent-review		 Separate Claude instance via claude -p. Zero session context. Independent perspective. Manual — you trigger /agent-review Independent code review with no bias. Security-sensitive code. High-stakes features. Parallel tasks. Routine chunk verification. Tasks where current session context is actually needed.
Skill
SKILL.md		 Instruction set Claude reads at task time. Gives Claude domain-specific knowledge. Auto — Claude reads when relevant Reusable domain knowledge. Style guides. Output formats. Things Claude should know consistently. Session-specific workflows. Things that change per project.
The Decision Rule Should it run automatically without prompting? → Hook. Should it run when I consciously decide to trigger it? → Command. Should it be done by a Claude with no context from this session? → Agent. Should Claude know how to do X consistently across all projects? → Skill.

❌ Common Mistakes

Using a command for something that should always run (use a hook).

Asking the current Claude to verify its own work without adversarial framing (use /verify or an agent).

Putting project-specific rules in a skill (put them in CLAUDE.md).

Building a hook for something that only matters sometimes (use a command).

✅ Correct Mapping

"Run type check after every file" → Hook (always, automatic).

"Review this code for security issues" → Agent (independent, unbiased).

"How to write a spec" → Skill (reusable, domain knowledge).

"Start a planning session" → Command (conscious phase transition).

03

Phase 1 · Plan

The five minutes that determine the quality of everything that follows.

The most expensive mistake in AI-assisted development is building the wrong thing quickly. Phase 1 exists to eliminate that. No code. No files. No implementation. Only questions, tradeoffs, and decisions.

When you type /plan, Claude shifts into architect mode. It asks seven structured questions about intent, scope, constraints, and existing opinions. It proposes an approach with a file map and the key decisions that must be resolved before building begins. For each decision, it presents options with honest tradeoffs and states what it would choose — and why.

You do not leave Phase 1 without a confirmed plan. The confirmed plan is the input to Phase 2.

When to Skip /plan Single-file changes. Obvious bug fixes. Trivial edits where the path is completely clear and no architectural decision is possible. If you are not sure whether to skip — do not skip. /plan costs five minutes. A wrong architectural decision costs sessions.
Phase 1 Flow 
/plan
↓ Claude asks: intent, why now, constraints, existing opinions
↓ You answer
↓ Claude proposes: approach + file map + key decisions
↓ Decisions resolved together (Claude states preference + reason)
↓ Confirmed plan — 5-7 bullet summary
→ Ready for /spec
04

Phase 2 · Spec

The contract that every future session honours.

A spec is not a requirements document. It is a constrained decision record that lives in the repository. Every Claude session that touches this feature reads it. It prevents scope drift, prevents re-litigation of settled decisions, and prevents a fresh Claude instance from making different architectural choices than the ones you agreed on in Phase 1.

When you type /spec, Claude asks for a name and generates /specs/[name].md using the structured template. The spec contains: intent, success condition, scope boundaries, technical approach, file map, locked decisions with reasons, and a chunked implementation checklist.

The checklist is the heartbeat of Phase 3. Each item is one committable unit of work. When a chunk is committed, the item is checked. The spec is the shared truth between all sessions and all phases.

Spec File — Key Sections 
## Scope
In Scope:  [explicit list — what WILL be built]
Out of Scope: [explicit list — what will NOT be touched]
            ↑ This section is as important as in-scope

## Decisions Locked
| Decision | Reason | Revisit When |
  ↑ Claude will not contradict these without flagging it first

## Implementation Checklist
- [ ] Chunk 1: [description]   ← One committable unit
- [ ] Chunk 2: [description]
  ↑ /chunk reads this. Each item becomes a /chunk session.
05

Phase 3 · Build

One chunk. One commit. Then stop.

Phase 3 is where most people lose discipline — and where most AI-generated code problems originate. The pattern is: ask Claude to build a lot in one go, accept the output without reading it carefully, skip the commit, ask for more. Three sessions later, the codebase is a tangle of partially-implemented ideas and nobody knows what was intentional.

The counter-pattern is simple: one chunk, then stop. /chunk identifies the next unchecked item from the spec, states exactly which files it will touch and which it will not, builds the chunk, and presents a commit message. You verify and commit. Then and only then does the next chunk begin.

The Two Hooks

post-edit.sh

Fires automatically after every file Claude writes or modifies. Runs type check, ESLint with auto-fix, and Prettier. Problems caught in zero seconds — before they pile up across 20 files.

Automatic · Zero tokens

pre-commit.sh

Fires before every git commit. Runs full type check and tests related to changed files. Blocks the commit if type errors exist. Last line of defence before code enters version control.

Automatic · Zero tokens
Scope Discipline If /chunk identifies that implementing the next item requires touching a file outside the spec's File Map, it stops and flags it. This is a feature, not a bug. Scope drift in Phase 3 is how a two-file feature becomes a twelve-file refactor. Every out-of-scope touch requires your explicit approval.
06

Phase 4 · Verify

Fast output is a risk signal, not a confidence signal.

The most dangerous moment in any Claude Code session is the moment after a clean-looking output. The hooks ran. No type errors. No lint issues. Everything looks fine. This is exactly when you need verification most — because the hooks caught syntax, not logic.

/verify makes Claude review its own output adversarially. It is not a standard review. Claude is asked to list every assumption it made, every way the code could fail in production, everything it added that wasn't asked for, and every edge case it did not handle. Then it gives a verdict: COMMIT / FIX FIRST / ESCALATE.

Two Verification Modes
/verify Standard chunk verification. 7 structured questions covering assumptions, failure modes, scope drift, type safety, and spec compliance. Returns a risk level and recommendation. Use after every chunk.
/verify full Full spec verification. All standard checks plus spec completeness, integration risk, and boundary testing. Use when all chunks are complete before marking a spec as done.
/agent-review Independent agent review. Spawns a fresh Claude instance via claude -p with the spec and git diff — but no session context. No familiarity bias. No attachment to decisions made. Use for security code, complex logic, anything going to production fast.
Why an Agent for Review The Claude that built the code has context bias. It knows why it made each decision. It will unconsciously interpret ambiguous code charitably. An agent spawned with only the spec and the diff has none of that bias. It reads the code exactly as a new engineer would read it during PR review — which is exactly what you need.
07

Phase 5 · Update

The compounding step. Every correction becomes a permanent improvement.

This is the phase most people skip. It feels optional because the feature is done. It is not optional. This is where the configuration layer gets smarter — where the investment you made in this session pays returns in every session that follows.

/update does four things: it harvests this session's corrections and patterns as CLAUDE.md candidates for your review. It closes the spec if the feature is complete. It checks if any reusable pattern was discovered. And it triggers the memory framework's /wrap to update SCRATCHPAD, DECISIONS, and SESSIONS.

The result: the next session opens with a CLAUDE.md that is smarter, a spec that is either closed or correctly up-to-date, and a SCRATCHPAD that puts Claude exactly where the work stopped.

The Compounding Mechanism Each /update session adds 1–3 rules to CLAUDE.md from real, observed corrections. After three months, your CLAUDE.md has 40–60 battle-tested rules. Claude produces consistently better output — not because the model changed, but because the configuration did. This is the return on the five minutes Phase 5 costs.

Full Session Flow

Every command, in order, with what triggers what.

Complete Feature Workflow 
── SESSION OPEN ────────────────────────────────────────────────
/resume          ← Memory Framework: reads SCRATCHPAD + DECISIONS
                    outputs Session Brief, waits for confirmation

── NEW FEATURE ─────────────────────────────────────────────────
/plan            ← Phase 1: architecture, decisions, confirmed plan
/spec            ← Phase 2: writes /specs/[name].md with checklist

── BUILD LOOP (repeat per chunk) ───────────────────────────────
/chunk           ← Phase 3: builds next checklist item
                    post-edit hook fires automatically on every file
                    (type check + lint + format)
/verify          ← Phase 4: adversarial self-review
git commit       ← pre-commit hook fires (type check + related tests)
                    chunk marked ✓ in spec
↑ repeat until all checklist items are ✓

── FEATURE COMPLETE ────────────────────────────────────────────
/verify full     ← Full spec verification
/agent-review    ← (optional) independent review for high-stakes

── SESSION CLOSE ────────────────────────────────────────────────
/update          ← Phase 5: harvest CLAUDE.md, close spec, wrap memory
                    triggers /wrap automatically
                    SCRATCHPAD + DECISIONS + SESSIONS updated

For mid-session decisions, use /decide from the Memory Framework immediately — do not wait for /update. Decisions captured at the moment of making are 10x more accurate than decisions reconstructed at session end.

Installation

Four steps. Works alongside Memory Framework and existing SCWA setup.

Step 1 — Copy commands 
mkdir -p /your-project/.claude/commands
cp .claude/commands/plan.md         /your-project/.claude/commands/
cp .claude/commands/spec.md         /your-project/.claude/commands/
cp .claude/commands/chunk.md        /your-project/.claude/commands/
cp .claude/commands/verify.md       /your-project/.claude/commands/
cp .claude/commands/update.md       /your-project/.claude/commands/
cp .claude/commands/agent-review.md /your-project/.claude/commands/
Step 2 — Enable hooks 
mkdir -p /your-project/.claude/hooks
cp .claude/hooks/post-edit.sh   /your-project/.claude/hooks/
cp .claude/hooks/pre-commit.sh  /your-project/.claude/hooks/
chmod +x /your-project/.claude/hooks/post-edit.sh
chmod +x /your-project/.claude/hooks/pre-commit.sh
cp .claude/hooks/pre-commit.sh  /your-project/.git/hooks/pre-commit
chmod +x /your-project/.git/hooks/pre-commit
Step 3 — Create specs directory + copy template 
mkdir -p /your-project/specs
mkdir -p /your-project/templates
cp templates/spec-template.md /your-project/templates/
Step 4 — Add to CLAUDE.md 
# Open CLAUDE-MD-ADDITION.md
# Copy the entire block between the ─── markers
# Paste into CLAUDE.md — after the Memory Framework block
# The ordering matters: Memory Protocol first, Phase Protocol second
