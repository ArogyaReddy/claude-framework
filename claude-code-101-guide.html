Claude Code 101

A professional summary of core principles, workflows, and best practices.

01 — Foundation

Plan Before You Prompt

The single most impactful habit when working with Claude Code is planning before typing. Activating Plan Mode (Shift + Tab twice) takes five minutes and consistently saves hours of downstream debugging. The quality gap between planned prompts and unplanned ones is not subtle — it is immediately visible in the output.

"Think first, then type. If you don't have a software engineering background, the fix isn't to skip planning — it's to have a real back-and-forth with the model, explore tradeoffs, and settle on a direction together."

Before requesting a feature, clarify the architecture. Before requesting a refactor, define the end state. Before debugging, articulate what is already known about the problem. Claude should be a collaborator in this process, not merely an executor.

02 — Prompting

Architecture Is the Prompt

Vague prompts hand Claude creative freedom it will use poorly. The more left unspecified, the more Claude fills in on its own — and that wiggle room is where most problems originate.

❌ Vague

"Build me an auth system."

✅ Specific

"Build email/password auth using the existing User model, store sessions in Redis with 24-hour expiry, and add middleware protecting all routes under /api/protected."

Three prompting principles that compound over time:

Specify constraints Tell Claude what not to do. "Keep this simple. One file if possible. No abstractions I didn't ask for."
Provide context "This runs on every request" or "this is a throwaway prototype" fundamentally changes which tradeoffs make sense.
Show examples When Claude misunderstands, write a minimal example of the desired output and say: "Apply this pattern to the rest."

If you are getting bad output from a capable model, the input is the problem. There is no shortcut around that.

03 — Configuration

The CLAUDE.md File

CLAUDE.md is the first thing Claude reads at the start of every session. Every instruction in it shapes how Claude approaches the entire project. It is onboarding material that runs before every conversation.

Keep it short Claude reliably follows ~150–200 instructions. Claude Code's system prompt already consumes ~50. Every instruction competes for attention — a bloated CLAUDE.md means Claude starts dropping rules unpredictably.
Make it specific Skip general knowledge Claude already has. Document the weird stuff: non-obvious bash commands, project-specific patterns, constraints you have encountered before.
Explain the why "Use TypeScript strict mode because we've had production bugs from implicit any types" is more useful than "use TypeScript strict mode." The reason helps Claude make judgment calls in unanticipated situations.
Keep it updated Press # during a session and Claude will append instructions to CLAUDE.md automatically. If you correct Claude on the same thing twice, it belongs in the file.
"Bad CLAUDE.md looks like documentation written for a new hire. Good CLAUDE.md looks like notes you'd leave yourself if you knew you'd have amnesia tomorrow."

04 — Context Management

Context Windows Degrade Before the Limit

Even with a 200,000-token context window, output quality begins to slip at roughly 20–40% usage. Compaction does not restore quality if the model was already degraded when compaction triggered.

🎯

Scope Conversations

One conversation per feature or task. Building auth and refactoring the database in the same session lets contexts bleed together.

📝

Use External Memory

Have Claude write plans and progress to SCRATCHPAD.md or plan.md. These persist across sessions so Claude can resume without starting from zero.

🔄

The Copy-Paste Reset

When context bloats: copy critical info from the terminal → /compact → /clear → paste only what matters. Fresh context, essential information preserved.

🧹

Know When to Clear

If a conversation has gone off the rails, /clear and start fresh. Claude still has CLAUDE.md — you are not losing project context.

The correct mental model: Claude is stateless. Every conversation starts from nothing except what you explicitly provide.

05 — Model Selection

Opus vs. Sonnet

There are real, measurable differences. Choosing the right model for the right task is a compounding efficiency gain.

Sonnet — Execution

Faster and more cost-effective. Excellent for tasks with a clear path: writing boilerplate, refactoring from a specific plan, implementing features where architectural decisions are already settled.

Opus — Reasoning

Slower and more capable. Better for complex reasoning, planning, and tasks requiring careful evaluation of tradeoffs before committing to a direction.

A durable workflow: use Opus to plan and decide on architecture, then switch to Sonnet (Shift + Tab) for implementation. Because both operate under the same CLAUDE.md constraints, the handoff is clean.

06 — Tooling

MCP, Hooks & Slash Commands

You don't need every feature immediately — but experimenting is worthwhile, because there is almost certainly something you are not using that would save real time.

MCP Servers Connect Claude to external services — Slack, GitHub, databases, APIs. If you are constantly pasting data from one tool into Claude, an MCP server can automate that entirely.
Hooks Run code automatically before or after Claude makes changes. Want Prettier on every file Claude touches? Hook. Type checking after every edit? Hook. This catches problems immediately instead of letting technical debt accumulate.
Slash Commands Create a .claude/commands/ folder and add Markdown files. Run them with /commandname. If you run the same kind of task repeatedly — debugging, reviewing, deploying — make it a command.

07 — Troubleshooting

When Claude Loops

When Claude tries the same failing approach repeatedly, the instinct is to add more instructions and more context. The better move is to change the approach entirely.

Clear the context Accumulated context may itself be the problem. /clear gives a fresh start.
Simplify the task Break complex tasks into smaller pieces. Get each piece working before combining. If Claude struggles with complexity, the plan going in was probably insufficient.
Show, don't tell Write a minimal example of the desired output yourself: "Here's what the output should look like. Now apply this pattern." Claude is very good at matching concrete examples.
Reframe the problem "Implement this as a state machine" versus "handle these transitions" can unlock completely different reasoning. The framing matters.

The meta-skill is recognizing a loop early. If you have explained the same thing three times and it has not landed, more explaining will not help. Change something.

08 — Advanced Usage

Build Systems, Not One-Shots

The practitioners extracting the most value from Claude Code are not using it for individual tasks. They are building systems where Claude is a component in a larger, automated workflow.

# Headless mode — scriptable, pipeable, chainable
claude -p "Review this PR and output a JSON summary" | jq '.issues'

Practical applications include automatic pull request reviews, support ticket responses, and documentation updates — all logged, auditable, and improvable over time.

"Claude makes a mistake → you review the logs → you improve CLAUDE.md or the tooling → Claude performs better next time. After months of iteration, the same models produce meaningfully better results — not because the models changed, but because the configuration did."

If you are only using Claude interactively, you are leaving compounding value on the table.
