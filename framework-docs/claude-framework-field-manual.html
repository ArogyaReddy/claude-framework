01

What It Is

Not a wrapper. Not a plugin. A persistent behavioral layer.

Claude Code sessions are stateless by default. Every morning you open a terminal, type claude, and it has no memory of what you were building, what decisions you made, what you tried that didn't work. You spend the first fifteen minutes re-explaining context. That's not pair programming — that's onboarding your own tool from scratch every day.

This framework ends that. It is a set of markdown files that Claude reads before your first message — a persistent briefing that gives Claude your identity, your project's rules, and an accurate summary of where you left off. The "intelligence" is the files. Claude just reads them.

Core Principle Move as much intelligence as possible into persistent context — CLAUDE.md, skills, hooks — so your per-task prompts become tiny, cheap, and precise. The best prompt is the one that requires the least explanation because everything structural is already loaded.

Zero Infrastructure

No server. No database. No build step. No API key beyond what Claude Code already uses. Markdown files on your hard drive.

Layered Non-Destructively

Drops into any existing project. Does not replace your stack, your tools, or your workflow. Adds a behavioral layer on top.

Two-Layer Safety

CLAUDE.md rules are layer one — behavioral. Hook scripts are layer two — mechanical. Layer two fires even when layer one drifts.

Cost-Efficient By Design

.claudeignore, skill invocation on demand, and session continuity typically cut per-session token spend 40–60%.

02

How It Works

What actually happens before you type your first message.

Before you type a single word, Claude Code has already loaded the framework into context. This is the actual sequence — not how it should work, but what is verified to happen every session.

Session Boot Sequence 
claude opens in project folder
  ↓
loads ~/.claude/CLAUDE.md     ← global policy: output rules, memory system, phase protocol
  ↓
loads ./CLAUDE.md            ← project policy: stack, conventions, active specs
  ↓
CLAUDE.md startup protocol fires:
  → reads PROFILE.md          ← who you are: role, stack, communication style
  → reads SESSION_LOG.md      ← where you left off: focus, decisions, next step
  → reads active specs       ← locked decisions for in-progress features
  ↓
you type your first message
Claude already knows everything. Zero re-explanation needed.
Important Distinction Rules in CLAUDE.md are not suggestions. They are visible text in Claude's working memory. Claude follows them the same way it follows its training. But they are not mechanical — they can drift as context fills. Hooks are mechanical. Both layers are necessary.
03

Memory System

Claude has no memory. Your hard drive does. The difference is a habit.

Claude Code does not store anything between sessions. No database. No vector store. Nothing in the model itself. When a session ends, context is wiped completely. The memory system works by having Claude write what happened to disk — and read it back next session. The "memory" is as good as what got written.

SCRATCHPAD.md
Rolling session state. Current focus, where to resume, failed attempts, open questions. Replaced entirely at each /wrap. Read at every session start. If this file doesn't exist, tomorrow's Claude starts blank.
DECISIONS.md
Permanent decision log. Never truncated. Every architectural choice, every rejected alternative. If a new decision contradicts an existing one, Claude flags it before writing. This file is your project's institutional memory.
SESSIONS.md
Lightweight audit trail. One row per session: date, summary, status. Never read at session start (noise). Written at /wrap only. Useful for project retrospectives and understanding the timeline of work.

Without /wrap

Session ends. Context wiped. Tomorrow's Claude: "I don't have context for this project. Could you explain what we're building?" You spend 15 minutes re-explaining. Every. Single. Morning.

With /wrap

Session ends. SCRATCHPAD.md written with exact focus, decisions, next step. Tomorrow: /resume → session brief in 10 seconds → working in 30 seconds. Same as walking back to your desk after lunch.

The One Non-Negotiable Rule Type /wrap before closing Claude Code. Every session, without exception. There is no technical enforcement of this — it must become reflexive. The cost of missing it is permanent loss of that session's context.
04

Skill Library

17 invocable behaviors. Each fires only when called. Zero idle token cost.

A skill is a markdown file containing precise behavioral instructions. When you say Use debug-first skill., Claude Code reads that file and injects the instructions into your session. The skill remains active for the conversation. No code runs. No compilation. Just text placed in front of Claude.

Skills cost tokens only when invoked. An unused skill costs nothing. This is why you don't auto-load all 17 at session start — that would burn 3,000–5,000 tokens before you've done anything.

debug-first
Use debug-first skill. — Forces EXPECTS → ACTUAL → CAUSE → FIX diagnosis before touching any code. The most important skill. Use it for every bug.
scope-guard
Use scope-guard skill. — Locks Claude to named files. Any attempt to touch anything else triggers SCOPE BLOCKER and Claude stops. Use before every multi-file task.
spec-to-task
Use spec-to-task skill on specs/[name].md — Converts a spec file to an ordered task list. One task per /chunk. Dependencies stated. Runs as subagent to protect your context.
code-review
Use code-review skill on [file]. — Reviews against CLAUDE.md rules. Returns violations only. No praise, no summary. Runs as subagent.
session-closer
Use session-closer skill. — Invoked by /wrap. Writes SCRATCHPAD.md and DECISIONS.md from full session context. The most consequential skill in the library.
healthcheck
Use healthcheck skill. — 12-point diagnostic: PASS / WARN / FAIL for every framework component. Run on any new project after framework-apply.
project-scan + framework-apply
Use project-scan skill on [path]. then Use framework-apply skill. — Scans for missing components, installs them. The onboarding sequence for any new project.
minimal-output
Use minimal-output skill. — Strips all narration. Code only. No preamble, no explanation, no process description. Maximum signal-to-noise for pure coding sessions.
Agent Flag Skills marked agent: true spawn a Claude Code subagent — they run in a separate context and don't consume your session's token budget. Use agent skills for file-heavy work (code-review, project-scan). Use agent: false skills for lightweight instruction injection.
05

Hook Scripts

Safety that Claude can't override. Fires at the app layer, not the context layer.

CLAUDE.md rules tell Claude not to do dangerous things. Claude follows them — until context drift, a long session, or an ambiguous prompt causes it to forget. Hook scripts fire at the Claude Code application layer, below Claude's context entirely. They cannot be overridden by anything Claude decides to do.

Three hooks. Three moments. All wired via ~/.claude/settings.json with absolute paths.

pre-tool-use.ps1
Fires before every Bash, Write, and Edit call.
Blocks: rm -rf, DROP TABLE, DROP DATABASE, TRUNCATE, git push --force, writes to .env, npm install, pip install, yarn add.
Logs every tool event to .claude/logs/tool-use.log.
post-tool-use.ps1
Fires after every tool call.
Logs outcomes, file paths of every Write/Edit, and failed Bash commands. Writes to tool-use.log, changes.log, and dated session-YYYY-MM-DD.log.
pre-compact.ps1
Fires before each /compact.
Saves state snapshot: files changed this session, git status, last SESSION_LOG entry. Writes to .claude/logs/pre-compact-state.md. Nothing is lost during context compression.
Critical: Use Absolute Paths Hook paths in settings.json must be absolute. A relative path works only when Claude Code is opened from the exact framework directory. Open it from anywhere else and the hook silently fails. Always: C:/Users/name/.claude/settings.json with forward slashes.
06

5-Phase Protocol

The difference between code that's fast to write and code that's right.

Without a protocol, AI-assisted development has one reliable failure mode: Claude starts writing before the architecture is clear, produces code that looks correct but has structural problems, and you spend more time fixing the structure than you would have spent planning. The code is fast to generate and slow to be right.

The 5-phase protocol applies to any feature touching more than two files. For single-file changes, /chunk + /verify is sufficient.

/plan — Phase 1
Architecture only. No code written. Claude proposes structure, resolves open decisions, identifies scope. You confirm or redirect. The plan must exist as text before Phase 2 begins.
/spec — Phase 2
Writes specs/[feature].md. The spec captures: scope, out-of-scope, locked decisions, constraints, and an ordered task checklist. One task per /chunk. This file is what tomorrow's Claude reads instead of your verbal re-explanation.
/chunk — Phase 3
Builds one bounded piece. One chunk = one commit-ready set of changes. Hooks fire on every file write. Output: file:line change manifest. Never batch multiple chunks without committing between them.
/verify — Phase 4
Claude interrogates its own output. Scope check (touched only named files?), logic check (matches the spec?), risk assessment. Returns: APPROVED or FLAGGED with reason. Run this before every commit. Always.
/update — Phase 5
CLAUDE.md improves. Every correction made during the session should become a rule. If Claude did something wrong that a rule could have prevented, write the rule. Then /wrap.
Why the Spec File Matters for AI Specifically Claude does not hold multi-session architectural intent in its head. If you plan on Monday and build on Tuesday without a spec file, Tuesday's Claude does not know Monday's plan. The spec is not documentation — it is the continuity mechanism.
07

The Session Loop

Open → work → close. Three habits that determine everything.
Daily Workflow 
── MORNING ──────────────────────────────────────────
cd /your-project && claude
/resume                  ← reads memory, outputs brief, waits for confirm
confirm the brief → start working

── DURING WORK ──────────────────────────────────────
Use scope-guard skill.   ← before any multi-file task
/plan                    ← non-trivial features: architecture first
/spec                    ← write the spec before touching code
/chunk                   ← build one piece
/verify                  ← interrogate before committing
/decide                  ← any significant decision: log it NOW
/compact                 ← at ~60% context, not 95%

── END OF DAY ───────────────────────────────────────
/wrap                    ← writes SCRATCHPAD.md + DECISIONS.md + SESSIONS.md
exit Claude Code
✓ /resume at session open — never skip, even for "quick tasks"
✓ scope-guard before every multi-file task
✓ /decide immediately when architecture decisions are made
⚠ /compact at 60% context, not when quality has already degraded
✗ Never close Claude Code without /wrap — there is no recovery
08

Token Discipline

Automatic savings vs. habit savings. Know which is which.

Some token savings are mechanical — they happen automatically once set up. Others require habits. Confusing the two is where cost discipline breaks down.

Automatic
.claudeignore — blocks node_modules, dist, logs. Saves 5,000–200,000 tokens per session automatically, once created.

CLAUDE.md auto-load — your rules and identity are in context before you type. Eliminates 300–500 tokens of re-explanation per session, every session.
Requires Habits
/wrap every session — SESSION_LOG saves re-explanation cost next session. Only works if you do it.

/compact at 60% — prevents quality degradation. Only works if you watch the meter.

scope-guard invocation — prevents file-sprawl (Claude reading unnecessary files for "context"). Only fires when invoked.

4-element prompts — SCOPE + ACTION + FENCE + OUTPUT reduces back-and-forth. Only when you write them this way.

Expensive Prompt

"Can you help me with the authentication? I need to add JWT validation and also maybe update the middleware and check the config file and make sure the tests still pass..."

4-Element Prompt

SCOPE: src/middleware/auth.js only.
ACTION: Add JWT validation to the checkToken function.
FENCE: Do not touch any other file.
OUTPUT: file:line changes only, no explanation.

09

Prompt Craft

The four elements. When FENCE and OUTPUT live in CLAUDE.md, you only write two.

Every effective prompt contains four elements. When FENCE and OUTPUT are defined in CLAUDE.md and skills, your prompts shrink to SCOPE + ACTION. That is the goal: prompts so short they can't be misinterpreted.

Scope

Exactly what file or function. Not "the auth module" — src/auth/middleware.js, the checkToken function.

Action

Exactly one thing being done. Not "refactor and improve" — "add try/catch around the fetchUser call."

Fence

What must not be touched. When scope-guard is active, this is enforced mechanically. Otherwise state it explicitly.

Output

Format, exclusions, stop condition. "file:line format only, no explanation, stop after writing the function."

Golden Prompt Templates 
── Bug diagnosis ──────────────────────────────────────
Use debug-first skill.
[Describe symptom — what happens vs. what should happen]

── Scoped code change ─────────────────────────────────
Use scope-guard skill.
SCOPE: src/auth/middleware.js — checkToken function only.
ACTION: Add JWT expiry validation.
OUTPUT: file:line changes only.

── Feature start ──────────────────────────────────────
/plan
Feature: user authentication — JWT + refresh tokens.
Constraints: no new dependencies, Lambda compatible.
Produce architecture only, no code.

── Code review ────────────────────────────────────────
Use code-review skill on src/api/users.js.

Setup & Verification

Install once. Verify with healthcheck. Apply to new projects in three commands.
First-Time Install 
# 1. Clone
git clone https://github.com/ArogyaReddy/claude-framework.git
cd claude-framework

# 2. Install (Windows)
.\setup.ps1

# 2. Install (Unix/Mac)
./setup.sh

# 3. Open Claude Code and verify
claude
Use healthcheck skill.     ← all 12 points should show PASS
Apply to Any Existing Project 
cd /path/to/your-project
claude
Use project-scan skill on .    ← produces gap analysis
Use framework-apply skill.     ← installs missing components
Use healthcheck skill.         ← verify: all PASS before real work
✓ CLAUDE.md present (both global ~/.claude/ and project root)
✓ PROFILE.md filled — no placeholder text remaining
✓ All 17 skills present in skills/
✓ Hook paths in settings.json are absolute with forward slashes
✓ .claudeignore present in project root
⚠ PowerShell execution policy: RemoteSigned or Unrestricted
⚠ SESSION_LOG.md will only exist after first successful /wrap
✗ Do NOT use relative paths in settings.json hook registration
