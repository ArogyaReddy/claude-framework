01

The Problem
This Solves

Claude Code has no memory. You pay for that every single session.

When a Claude Code session closes, everything is gone. The next session opens blank. Claude knows your codebase only because it can read files. It knows your decisions, your constraints, your failed attempts, your current focus — only if those things are written somewhere it can read.

Without a memory system, you spend the first 5–10 minutes of every session re-establishing context. You explain the architecture again. You mention the constraint again. You warn Claude about the approach that already failed. You cost yourself 3,000–6,000 tokens per session just to get back to zero — and that is before any real work begins.

Multiply that by 5 sessions a week. That is 15,000–30,000 tokens a week spent on re-explanation. That is real cost. And it compounds — because without a decision log, the same architectural decisions get re-litigated. Without a scratchpad, the same failed approaches get suggested again. Without a session trail, you lose track of why the code is the way it is.

Core Insight The memory problem is not a Claude limitation to work around. It is a configuration problem to solve once. A three-file system on disk costs ~1,300 tokens per session pair and eliminates the re-explanation tax permanently.

This framework solves it completely. Three plain Markdown files. Three slash commands. One optional hook. No external services. No API keys. Everything is version-controllable, readable, and portable.

02

Architecture Overview

Three files. Three commands. One protocol.

Each file has a single, non-overlapping responsibility. They do not duplicate each other. Keeping them separate is the design — if you merge them, you lose the ability to control what Claude reads and when.

SCRATCHPAD.md Rolling current state. What you're working on, where to resume, what failed, what's unresolved. Replaced and updated each session. Kept under 300 lines. This is the file Claude reads first.
DECISIONS.md Permanent decision log. Every architectural or significant decision, with its reason and the conditions under which it should be revisited. Never truncated. Never deleted. Append only. This is your institutional memory.
SESSIONS.md Audit trail. One row per session — date, focus, outcome, next step. Claude does not read this at session start (too much noise). You use it when you need to answer "when did we change X?" or "what happened last Tuesday?"
Session Flow

Session Opens

Run /resume. Claude reads SCRATCHPAD + relevant DECISIONS. Outputs brief.

→

Work Happens

Use /decide when decisions are made. Claude flags DECISIONS.md conflicts.

→

Session Closes

Run /wrap. All three files updated. Resume point written. CLAUDE.md suggestions surfaced.

→

Next Session

/resume reads the updated files. Claude is fully oriented in 30 seconds. No re-explanation.

03

The Three
Memory Files

What each file contains and why it is structured that way.

SCRATCHPAD.md

The session state file. Think of it as the shift handover note — the minimum a fresh person needs to resume without losing ground. Not a log. Not a journal. A rolling briefing document.

SCRATCHPAD.md — Structure 
## What We're Building
Written once. Updated only when scope changes. The "why" as much as the "what".

## Current Focus
Replace entirely each session. The one specific thing being worked on.
With: why it matters + what is explicitly out of scope this session.

## Resume Here
→ ONE sentence. Specific file path + line number if applicable.
   A fresh Claude instance should know exactly what to do.

## Active Files
3-6 files most relevant to current focus. Speeds up orientation.

## What Was Tried and Failed
Append only. Never delete. Prevents Claude re-suggesting dead ends.
Format: [date] — what was tried → why it failed

## Open Questions
Unresolved things. Claude knows they exist but does not try to resolve
them without new information.

## Context Claude Needs
Business constraints, client requirements, team decisions not in the codebase.
Size Rule Keep SCRATCHPAD.md under 300 lines. When it grows beyond this, the /wrap command will summarise older "Failed" entries into a paragraph. A bloated scratchpad degrades the value of reading it — if Claude has to read 600 lines to orient, you have lost the efficiency gain.
DECISIONS.md

The institutional memory file. This is the one that pays the longest dividends. Six months from now, when you or a new developer touches a piece of code and asks "why is this done this way?" — the answer is in DECISIONS.md, with the reasoning that was compelling at the time.

DECISIONS.md — Entry Format 
### [DECISION-007] Redis for session storage, not database
Date: 2026-03-18
Status: Active
Decision: Store auth sessions in Redis, not Postgres.
Reason: p99 latency requirement of <10ms from Client X SLA.
         DB reads were consistently 35-80ms under load.
         This is a contractual constraint, not a preference.
Alternatives Considered: Postgres (too slow), in-memory (no horizontal scale).
Revisit When: Client X SLA changes, or Redis hosting cost exceeds $400/mo.

The reason field is the most important field. A decision without its reason is a constraint that looks arbitrary. An arbitrary-looking constraint gets violated. A well-reasoned constraint gets respected — by Claude, by new developers, by you six months from now when you have forgotten why you made it.

SESSIONS.md

One row per session. Append only. Five words per column. Claude does not read this at session start — it is for your benefit, not Claude's. Use it when you need to reconstruct the history of a decision or understand why the codebase looks the way it does.

SESSIONS.md — Row Format 
| Session | Date       | Focus                  | Outcome                    | Next               |
| 001     | 2026-03-15 | Auth webhook handler   | Redis lock implemented     | Write tests        |
| 002     | 2026-03-17 | Webhook tests          | 4 tests passing, 2 failing | Fix race condition  |
| 003     | 2026-03-18 | Race condition fix     | SET NX pattern works       | PR and review      |
04

The Three
Commands

One keystroke each. The entire memory protocol in three triggers.

/resume — Session Start

Reads SCRATCHPAD + relevant DECISIONS. Outputs a Session Brief. Waits for your confirmation before starting work. Use at the start of every session without exception.

/wrap — Session End

Updates all three memory files. Surfaces CLAUDE.md suggestions for your review. Writes the resume point. Use at the end of every session. Takes 30 seconds. Saves 5 minutes next time.

/decide — Mid-Session

Logs a decision to DECISIONS.md immediately — while context is fresh. Do not wait for /wrap. Decisions recorded at the moment of making are 10x more valuable than decisions reconstructed at session end.

What /resume Outputs
Session Brief — Example Output 
## Session Brief

Resuming: Fixing the race condition in Stripe webhook idempotency handler.

Next action: → Implement Redis SET NX pessimistic lock in
              /api/webhooks/stripe.ts line 47, replace current optimistic lock.

Watch out for: - Optimistic locking was tried (session 002) — race condition persisted.
               - Queue-vs-drop decision still open — waiting on product sign-off.

Active constraints: - No new dependencies without approval (CLAUDE.md)
                    - All DB access via /lib/db — not applicable here (Redis direct)

Decisions that apply: - DECISION-007: Redis for sessions (<10ms latency req)
                       - DECISION-012: Stripe webhooks only, no Paddle

Ready to proceed. Confirm or redirect.

That brief costs ~700 tokens to generate. It replaces 5 minutes of re-explanation. Every time.

Daily Session Flow

The ritual that makes everything else work.

❌ Without Framework

Open session.
Re-explain project context (2 mins).
Re-explain current focus (1 min).
Re-explain what failed last time (1 min).
Re-establish constraints (1 min).
5 minutes lost. 4,000 tokens wasted.
Claude still makes the same mistakes.
Same decisions re-litigated.
No record of why code is the way it is.

✅ With Framework

Open session.
Type: /resume
Read Session Brief (30 seconds).
Type: confirmed, let's go
Working in under 2 minutes.
700 tokens spent. Zero re-explanation.
Failed approaches blocked automatically.
Decisions respected without repeating them.
Full institutional memory intact.

The Full Daily Checklist
▸ Session start: Run /resume before typing anything else. Confirm the brief.
▸ When a decision is made: Run /decide immediately. Do not wait for session end.
▸ When Claude contradicts a decision: Claude should flag it. If it doesn't, check DECISIONS.md yourself before accepting the output.
▸ Before closing: Run /wrap. Do not close the session without updating memory files. The next session's quality depends entirely on this step.
▸ Never skip /wrap thinking "I'll remember tomorrow." You won't. Claude definitely won't. The entire framework fails if wrap is inconsistent.

Token Cost Analysis

The cheapest thing in your workflow. By a significant margin.

Every token spent on the memory framework displaces a larger token cost elsewhere. This is not a cost to minimise — it is an investment to make deliberately, because the return ratio is approximately 7:1 per session pair.

Action Token Cost Tokens Saved Net
/resume at session start ~700 ~5,000 (re-explanation) +4,300
/wrap at session end ~650 ~5,000 (next session) +4,350
/decide mid-session ~200 ~2,000 (future re-litigation) +1,800
Per session pair (typical) ~1,550 ~12,000 +10,450 (~7:1 ratio)
Compounding Return The DECISIONS.md savings compound over time. A decision logged in month 1 prevents re-litigation in month 3, month 6, and month 12. The token savings on a single well-documented decision can exceed 10,000 tokens across the lifetime of a project — from a 200-token investment.
Git Strategy Add SCRATCHPAD.md and SESSIONS.md to .gitignore — they are local session state. Keep DECISIONS.md in version control — it is institutional memory that the whole team benefits from.
File Size Control SCRATCHPAD.md: 300-line cap, enforced by /wrap. DECISIONS.md: no cap — never truncate. SESSIONS.md: grows ~2 lines/week — negligible. Total read cost at session start: ~600-900 tokens.
Sonnet vs Opus Run /resume and /wrap on Sonnet. These are structured read/write tasks, not reasoning tasks. Save Opus for architectural decisions — which you then immediately log with /decide.

Installation

Five steps. Under ten minutes. Works alongside your existing setup.

Step 1 — Copy memory files to project root 
cp SCRATCHPAD.md     /your-project/
cp DECISIONS.md      /your-project/
cp SESSIONS.md       /your-project/
Step 2 — Copy commands to .claude/commands/ 
mkdir -p /your-project/.claude/commands
cp .claude/commands/resume.md   /your-project/.claude/commands/
cp .claude/commands/wrap.md     /your-project/.claude/commands/
cp .claude/commands/decide.md   /your-project/.claude/commands/
Step 3 — Install hook (optional — auto-wrap on close) 
mkdir -p /your-project/.claude/hooks
cp .claude/hooks/post-session.sh /your-project/.claude/hooks/
chmod +x /your-project/.claude/hooks/post-session.sh
Step 4 — Add memory protocol to your CLAUDE.md 
# Open CLAUDE-MD-ADDITION.md
# Copy the entire block between the ─── markers
# Paste into your existing CLAUDE.md
# This is what makes Claude follow the protocol automatically
Step 5 — Initialise SCRATCHPAD.md 
# Fill in ONE section manually:
# "What We're Building" — one paragraph, the project purpose
# Everything else gets written by Claude via /wrap

# Then run your first session:
/resume    ← Claude will note the scratchpad is fresh and ask what to focus on
Compatibility This framework is additive. It does not replace or modify anything in your existing SCWA setup — no existing hooks, no existing CLAUDE.md rules, no existing specs. It layers on top. The only file you edit is CLAUDE.md, and only by appending the memory protocol block.
08

Maintenance Rules

The rules that keep the framework working six months from now.

▸ SCRATCHPAD.md: 300-line cap. /wrap enforces this. If you see it growing between wraps, summarise "Failed" entries manually before the next session.
▸ DECISIONS.md: never truncate, never delete. If a decision is reversed, mark it [REVERSED] and add a new entry explaining why. The reversal is itself a decision worth recording.
▸ SESSIONS.md: append only via /wrap. Never edit manually. Its value is in being a reliable, untampered audit trail.
▸ CLAUDE.md: /wrap surfaces suggestions, never auto-edits. You review every CLAUDE.md change. This is intentional — permanent project constraints require human judgment, not automatic promotion.
▸ If Claude contradicts a DECISIONS.md entry: the entry wins until you explicitly decide to reverse it. Claude should flag the conflict — if it doesn't, that is a CLAUDE.md addition: "Always flag before contradicting a DECISIONS.md entry."
▸ /wrap is non-negotiable at session end. One skipped wrap means the next session starts with stale memory. Two skipped wraps means the scratchpad is meaningfully inaccurate. Three means you have abandoned the framework.
The Long Game At six months, your DECISIONS.md has 30–50 entries, each with its reason. Your SCRATCHPAD.md always reflects exactly where you are. Your sessions open in under two minutes. The same architectural mistakes are never made twice. That is not a vision — it is what this framework produces if you run /resume and /wrap consistently. The entire investment is two keystrokes per session.
