01

The Right Mindset

Before your fingers touch the keyboard.

Most people approach Claude Code as a faster way to Google. That is the wrong mental model and it explains most of the disappointing results people report.

Claude Code is a collaborating principal engineer — one that is extraordinarily fast, has no ego, works at 3am, and will follow your architectural decisions without argument. But like any engineer, what it builds is only as good as the brief it receives.

The mindset shift that changes everything: you are the architect. Claude is the implementation arm. The moment you invert that — the moment you ask Claude to be the architect — you inherit all of its defaults, its pattern-matching to common codebases, its tendency to over-engineer. You get a system built for a project Claude imagined, not the one you are actually building.

Core Principle Claude is stateless by nature. Every conversation begins from nothing except what you explicitly hand it. You are not working with a colleague who remembers context. You are working with the world's best contractor who reads a brief on arrival.

This is not a limitation to work around. It is a design constraint to work with. When you internalize it, you stop getting frustrated that Claude "forgot" something. You start building briefs — CLAUDE.md, plan files, scratchpads — that make the statelessness irrelevant.

Professionals who get the most out of Claude Code think like systems designers: they invest in the configuration layer once, and collect returns on every session thereafter.

02

Session Architecture

The five minutes that buy you hours.

There is a ritual that separates professionals from beginners, and it costs five minutes.

Before every session of meaningful work, activate Plan Mode. Hit Shift + Tab twice. Do not type the first real prompt until you have planned.

Key Binding 
Shift + Tab + Tab    →  Enter Plan Mode
Shift + Tab           →  Toggle Sonnet ↔ Opus
#                    →  Append to CLAUDE.md mid-session
/compact             →  Compress context to summary
/clear               →  Full context reset (keeps CLAUDE.md)
/cost                →  Token spend for current session

In Plan Mode, have Claude map what you are about to build before it writes a line. Ask it for tradeoffs. Push back. Settle on a direction together. The quality gap between planned sessions and unplanned ones is not subtle — it shows up in file count, abstraction depth, and the ratio of first-pass code that actually runs.

The session structure I use for any non-trivial feature:

Phase 1 · Plan Plan Mode. Describe intent. Ask Claude what it needs to know. Review its proposed approach. Correct architecture before a single file is created.
Phase 2 · Spec Have Claude write a plan.md with agreed approach, file map, and decisions. This becomes external memory for the session and the next one.
Phase 3 · Build Switch to Sonnet. Implement per the spec. One logical chunk at a time. Commit after each chunk works.
Phase 4 · Verify Ask Claude to interrogate its own output. "What assumptions did you make? What could break? What edge cases did you skip?" Then read the code yourself.
Phase 5 · Update Press # and update CLAUDE.md with anything you corrected or decided. Make the next session smarter than this one.
03

CLAUDE.md —
Your Constitution

The document that runs before everything else.

CLAUDE.md is the first file Claude reads at the start of every session. It shapes every decision that follows. Treat it like the constitution of your project — clear, authoritative, minimal, and frequently amended.

Most teams either skip it entirely or fill it with noise. Both are costly mistakes.

Hard Constraint Claude reliably tracks ~150–200 active instructions. Claude Code's own system prompt consumes roughly 50 of those slots. Every line in CLAUDE.md competes for attention. A bloated file means Claude starts silently dropping rules — and you won't know which ones.
Keep It Short Target under 100 meaningful lines. Cut anything Claude already knows from training. No folder explanations. No general coding advice.
Project-Specific Only Document the weird stuff: non-obvious bash commands, constraints from past bugs, naming conventions that deviate from the norm, external services and their quirks.
Always Explain Why "Use TypeScript strict mode" is weak. "Use TypeScript strict mode — we had three production bugs from implicit any in Q3" gives Claude the context to extend that rule to situations you didn't anticipate.
Update Every Session Every time you correct Claude on the same thing twice, that correction belongs in CLAUDE.md. Press # mid-session. The file should grow smarter with every week you use it.
Section for SDD If you run a spec-driven workflow, include a section declaring your spec format, where specs live, and which specs are active. Claude will honour them without being reminded each time.
CLAUDE.md Structure 
# Project Constitution

## Stack
Next.js 14 · TypeScript strict · Postgres · Redis

## Constraints
- No new dependencies without asking first
- All DB queries go through /lib/db — never raw SQL in routes
- Auth: we use our own JWT, not next-auth (legacy decision, do not refactor)

## Commands That Matter
- npm run check:types → must pass before any commit
- npm run seed → resets local DB to fixture state

## Active Specs
- /specs/auth-refresh.md      (in progress)
- /specs/billing-v2.md        (planned)

## Hard Rules
- Never use `any`. If you must, comment why above the line.
- Keep components under 200 lines. Split if larger.
04

Prompting Like
an Architect

Precision is not pedantry. It is respect for your own time.

Bad output from a capable model always traces back to the input. There is no exception to this rule. The model is not broken. The brief was incomplete.

❌ What Not to Do

"Build me an auth system"

"Refactor the user module"

"Make this faster"

"Fix the bug"

✅ What to Do

"Build email/password auth using the existing User model, Redis sessions with 24h TTL, middleware on /api/protected only."

"Refactor UserService to use the repository pattern. Keep the same public interface — callers must not change."

"This runs on every request. Optimise for the hot path. No new dependencies."

"Here is the stack trace. Here is what I already tried. Here is my hypothesis."

Prompt Principles

First Message Sets Trajectory

Claude anchors on your first message. Thirty extra seconds invested there compounds through the entire session. Never rush the opener.

Constrain the Solution Space

Tell Claude what you do not want. "Minimal. One file if possible. No abstraction I didn't ask for." The absence of constraints is an invitation to over-engineer.

Give Context About Why

"This is a prototype we'll throw away" and "This runs in production at 10k rpm" require different tradeoffs. Claude cannot infer what you haven't stated.

Show Before You Tell

When Claude keeps misunderstanding, write a minimal example of the desired output. "Apply this pattern to the rest." Pattern-matching from a concrete example almost always works where prose fails.

The Verification Prompt

After Claude builds something, ask: "What assumptions did you make? What edge cases aren't handled? What could break under load?" It surfaces issues it wouldn't volunteer.

Name the End State

Before refactors: "The end state should look like X." Before debug: "Success means the test at line 47 passes." A clear exit condition prevents scope drift.

05

Context is a Resource

Manage it like memory. Spend it intentionally.

A 200,000-token context window does not mean you get 200,000 tokens of consistent quality. Degradation begins around 20–40% usage. The model starts losing the thread — subtly at first, then obviously. By the time you notice, you are already in a hole.

Compaction does not restore quality. If the model was degraded when /compact triggered, you get a degraded summary. The summary then continues to degrade. Many engineers experience this and assume the model is bad. The model is fine. The session was mismanaged.

One Task Per Session One conversation per feature or task. Building auth and then refactoring the database layer in the same session bleeds contexts together. Separate them.
External Memory Always For any work spanning more than one session, have Claude write a SCRATCHPAD.md or plan.md. These persist. Claude reads them on arrival next session and resumes without losing ground.
The Copy-Paste Reset When context bloats: copy critical state from terminal → /compact → /clear → paste only what matters. Fresh context. Essential information preserved. This is not defeat — it is maintenance.
Monitor With /cost Run /cost periodically. When token spend looks high relative to what was produced, consider resetting before quality drops. Prevention is cheaper than recovery.
Clear Without Guilt /clear does not erase project context. Claude still has CLAUDE.md. Nine times out of ten, clearing a degraded session and starting fresh produces better output than continuing a polluted one.
06

Model Selection Strategy

The right tool for the right decision type.

The real question is not "which model is smarter" — it is "what is the cost of a wrong decision here?"

When a wrong decision cascades through architecture — a bad data model, a broken auth assumption, a misunderstood business rule — you pay for it across dozens of subsequent sessions. When a wrong decision is local and cheaply verified — a loop written incorrectly, a utility function that fails a test — you catch it immediately and fix it in thirty seconds.

Calibrate model weight to decision reversibility.

Opus — High-Stakes Reasoning

Architecture decisions. Security design. Data modeling. Performance tradeoffs. Anything where a mistake silently propagates. Anything requiring careful evaluation of edge cases before committing.

Sonnet — Execution

Implementing a plan that already exists. Writing boilerplate. Refactoring within defined constraints. Running tests and fixing failures. Any task where the path is clear and output is verifiable immediately.

Workflow Use Opus to plan and decide. Switch to Sonnet (Shift + Tab) for implementation. Your CLAUDE.md ensures both models operate under the same constraints — the handoff is clean. This alone can cut session costs by 40–60% on implementation-heavy work.
07

Git Discipline
With AI Code

Your recovery tool. Not an afterthought.

This section does not appear in most Claude Code tutorials. It should be the first thing they cover.

AI-generated code moves fast across many files simultaneously. Claude will cheerfully modify 40 files in a single response if the task warrants it. Without disciplined git hygiene, untangling a bad batch of changes becomes a forensic exercise.

Hard Rule Commit before you ask Claude to make significant changes. If Claude produces a bad output, git restore . returns you to a clean state in one command. Without a clean commit, you are manually unpicking changes across dozens of files.
▸ Commit before every significant Claude request. Treat each Claude session phase as a unit of work with a clean entry and exit point.
▸ Commit after every working chunk. When Claude completes a piece and it works, commit before asking for the next piece. Never build on unverified foundation.
▸ Use descriptive commit messages. When reviewing AI sessions, you need to know what Claude was asked to do at each step. The commit message is the audit trail.
▸ Review git diff before committing. AI-generated code is fast. It is also sometimes subtly wrong. Read what changed. Every time.
▸ Use branches for features. Never build directly on main with AI assistance. Branch per feature. Merge when verified. This contains blast radius.
▸ Never blindly accept multi-file changes without reading them. The output speed of Claude Code creates an illusion of trustworthiness. Fast is not the same as correct.
08

Spec-Driven Development

The architecture layer that makes everything else work.

If you have worked with Claude Code for any length of time, you have noticed a pattern: the quality of the output correlates almost entirely with the quality of the intent you expressed going in. Spec-Driven Development (SDD) is the practice of making that intent explicit, versioned, and persistent — before you prompt.

A spec is not a requirements document. It is a constrained decision record: what you are building, what architectural decisions were made and why, and what the boundaries are. It lives in your repository. Claude reads it. It informs every prompt in the feature's lifetime.

Spec File Structure 
# /specs/auth-refresh.md

## Intent
Implement JWT refresh token rotation. Access tokens expire
in 15 minutes. Refresh tokens expire in 30 days and rotate
on use (old token invalidated on exchange).

## Decisions Made
- Storage: Redis (not DB) — latency requirement <5ms per check
- No refresh token families: simple rotation only
- Invalidation: DELETE key, not blacklist

## Boundaries
- Do not touch User model — it is owned by /specs/user-v3.md
- Middleware must be composable, not monolithic

## Status
[ ] Refresh endpoint   [ ] Middleware   [ ] Tests

The payoff is compounding. Today's spec decisions become tomorrow's CLAUDE.md additions. After six months, your project has a documented decision history that any new developer — or any new Claude session — can read and immediately understand. You stop re-explaining the same architecture in every session.

The implementation detail: reference active specs in CLAUDE.md so Claude knows to honour them without being reminded. Closed specs stay in the repo as read-only history. Claude won't break them accidentally if it knows they're closed.

SDD Principle A spec is not bureaucracy. It is the difference between Claude making decisions in your context versus Claude making decisions in the context of a similar open-source project it saw in training. The spec is how you ensure the former.
09

Diagnosing Failure Loops

When to push. When to pivot. When to start over.

At some point Claude will loop: trying the same failing approach, incrementally adjusting, failing again. Or it will produce something confidently wrong and be unmoved by your corrections. The instinct is to add more instruction. This is almost always wrong.

If you have corrected the same thing three times and it has not landed, more correcting will not help. Something about the approach — the framing, the context, the task scope — needs to change.

Clear First Run /clear. The accumulated context may itself be the problem. A fresh session with a sharper first message often resolves in one response what the degraded session could not in twenty.
Shrink the Task If Claude cannot handle the full problem, extract the smallest piece it can succeed at. Get that working. Build back up. If Claude struggles with a complex task, the plan was insufficient — not the model.
Show, Don't Tell Write a two-line example of exactly what you want. "This is the shape. Now produce this for the full dataset." Claude's pattern-matching from a concrete example outperforms its comprehension of abstract prose instructions.
Reframe the Problem "Implement this as a state machine" versus "handle these transitions" can unlock entirely different reasoning paths. The mental model you impose in the prompt shapes how Claude reasons about the solution.
Blame the Brief Before blaming the model, re-read your original prompt. Almost every persistent failure traces back to an ambiguity in the first message that Claude resolved in a way you didn't expect.

Verification is
Non-Negotiable

Fast is not the same as correct. Ever.

Claude Code produces output at a speed that creates an illusion of authority. This is the most dangerous part of the tool for engineers who are still building their instincts. You must read the code Claude writes. Every file. Every time.

Claude — particularly in fast execution mode — has a pattern of overengineering: extra files, unnecessary abstraction layers, defensive code for cases you never mentioned. You can end up with twelve files for something that needed two lines if you do not check the output against the intent.

Verification Workflow After every significant Claude output: run the verification prompt. "What assumptions did you make in building this? What edge cases are not handled? What would break under production load? What did you add that I didn't ask for?" These four questions surface the vast majority of issues before they reach your tests.
▸ Read git diff before accepting any multi-file change.
▸ Run your type checker after every Claude session: npm run check:types or equivalent.
▸ Count the files Claude created. If the number surprises you, investigate before moving on.
▸ Check for added dependencies in package.json. Claude will install things without announcement.
▸ Verify that Claude did not modify files outside the scope of the task. It happens.
▸ Never accept security-relevant code — auth, encryption, session handling — without reading every line yourself. Speed does not justify blindness here.

Hooks help here significantly. See Chapter 11. A type-check hook and a linter hook that run on every file Claude touches catch the most common class of issues automatically — without discipline fatigue.

11

Hooks, MCP
& Commands

The infrastructure layer beneath the prompts.

You do not need every feature. But you should treat this layer as the multiplier on everything else. One well-configured hook or MCP server eliminates a class of manual work permanently.

Hooks

Hooks run code automatically before or after Claude makes changes. They are the most underused feature in Claude Code and among the highest-leverage ones.

.claude/hooks/post-edit.sh 
#!/bin/bash
# Runs after every file Claude writes or modifies

# Auto-format changed files
prettier --write "$CLAUDE_FILE_PATH"

# Type-check immediately
tsc --noEmit --skipLibCheck

# Lint the changed file only
eslint "$CLAUDE_FILE_PATH" --fix

Problems caught by a hook are caught in zero seconds, before they accumulate. Technical debt from a long AI session is a real phenomenon — hooks are the primary countermeasure.

MCP Servers

If you are manually copying information from an external tool into Claude, there is almost certainly a Model Context Protocol server that eliminates that step. MCP servers connect Claude directly to Slack, GitHub, Jira, databases, internal APIs — anything that exposes structured data.

The test: count how many copy-paste steps exist in your current workflow between the tool and Claude. Each one is a candidate for an MCP server. Even if no server exists for your tool, building one is a matter of exposing structured data over a standard interface.

Custom Slash Commands
.claude/commands/review.md 
# Triggered with: /review

Review the changes in the current session against the active spec.

Check for:
- Scope creep (files or abstractions not in the spec)
- Missing edge cases from the spec's boundary conditions  
- Type safety violations
- Anything added that was not explicitly requested

Output a short list of findings. Flag anything that should be 
reverted before this moves to review.

If you run the same kind of task more than three times in a week — review, debug, document, deploy prep — make it a command. The activation cost is ten minutes. The return is every subsequent instance of that task running in one keystroke.

12

Build Systems,
Not Sessions

The compounding loop that separates good from great.

The practitioners extracting the most from Claude Code are not using it interactively for individual tasks. They are building systems in which Claude is a component — scheduled, scripted, integrated into workflows that run without human intervention.

Headless Mode · -p Flag 
# Run a prompt without entering interactive mode
claude -p "Review this PR diff and output a JSON summary of concerns" \
  < pr-diff.txt \
  | jq '.concerns[] | select(.severity == "high")'

# Pipe into documentation update
git diff HEAD~1 \
  | claude -p "Update CHANGELOG.md based on this diff" \
  >> CHANGELOG.md

The headless flag unlocks Claude Code as a pipeline component. Automatic PR reviews. Automatic changelog generation. Automatic test gap analysis on new code. All logged, auditable, and improvable.

The Compounding Loop Claude makes a mistake → you review the logs → you update CLAUDE.md or the tooling → Claude performs better next time. After months of this loop, the same underlying models produce meaningfully better output than at launch — not because the models improved, but because the configuration did. This is the return on investment that most people never reach.

The discipline required: treat every correction as a system update, not a one-off fix. Every time you type the same correction twice, that correction should have been in CLAUDE.md. Every time you manually run the same verification step, that step should be a hook. Every time you paste the same context block, that context should be in a plan file.

The engineers who use Claude Code at its ceiling are not smarter about AI. They are more disciplined about configuration. The ceiling is high. Most people never find it because they are too busy prompting to invest in the infrastructure that makes prompting compoundingly better.

The End State
Session opens Claude reads CLAUDE.md. Reads the active spec. Reads SCRATCHPAD.md. Knows the codebase constraints and the current task without a single explanatory message from you.
Work happens Hooks run on every edit. Types are checked. Linting is automated. You review diffs, not files. You read output, not prompts.
Session closes SCRATCHPAD.md is updated. CLAUDE.md gets any new rules. Commit is clean. The next session starts smarter than this one did.

That is not a vision. That is achievable in two weeks of disciplined setup. The investment is front-loaded. The returns are indefinite.
